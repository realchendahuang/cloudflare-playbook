## 先判断

| 场景 | 建议 |
| --- | --- |
| 公开配置、功能开关 | 优先用 KV。 |
| 接口响应缓存 | 可以用，前提是允许短时间旧数据。 |
| 公开目录、静态索引、清单文件 | 很适合，构建期或后台任务写入。 |
| 用户偏好 | 可以用；强撤销和权限校验要谨慎。 |
| 登录会话 | 不优先。 |
| 评论、订单、文章关系数据 | 不适合，交给 D1。 |
| 计数器、库存、排行榜 | 不适合，交给 Durable Objects 或 D1。 |
| 文件、图片、大对象 | 不适合，交给 R2。 |

## 成本和一致性边界

| 边界 | 判断 |
| --- | --- |
| 读取 | 页面减少散读，能合并就合并成清单。 |
| 写入 | 适合低频后台写入，不适合评论、订单、计数器和状态流。 |
| 删除和列举 | 用户请求路径避免扫描命名空间，分页和筛选交给 D1。 |
| 存储内容 | 大对象、附件和图片进 R2。 |
| 同一个键写入 | 付费也不会把同键高频写入变成强项。 |
| 一致性 | 权限撤销、订单确认、库存和锁需要 D1 或 Durable Objects。 |

## 常见坑

| 坑 | 为什么危险 | 怎么避开 |
| --- | --- | --- |
| 把 KV 当强一致数据库。 | 其他地区可能短时间读到旧值。 | 权限撤销、库存、订单确认用 D1 或 Durable Objects。 |
| 同一个键高频写入。 | 超过每秒 1 次写入会失败。 | 拆键，或用 Durable Object 集中写入。 |
| 每个请求读很多散键。 | Free 读额度很快被打满。 | 合并配置，预生成清单。 |
| 用户请求里列举全部键。 | 列举操作单独计费，也不适合业务分页。 | 查询和分页交给 D1。 |
| 把附件塞进 KV。 | 体积、成本和读取方式都不合适。 | R2 存文件，D1 / KV 存索引。 |
| 本地、预览、生产共用命名空间。 | 误写后排查困难。 | 按环境拆命名空间。 |

## 和其他产品怎么选

| 目标 | 更适合 |
| --- | --- |
| 读多写少配置、缓存、公开索引 | KV |
| 关系查询、排序、筛选、事务 | D1 |
| 文件、图片、附件、大对象 | R2 |
| 单实体强一致、写冲突、连接状态 | Durable Objects |
| 异步更新缓存、削峰、重试 | Queues |
| HTTP 层缓存 | Cache Rules / CDN Cache |
