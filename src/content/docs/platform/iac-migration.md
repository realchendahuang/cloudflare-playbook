title: 迁移与配置管理
---

## 适用场景

| 场景 | 处理方式 | 首要操作 |
| --- | --- | --- |
| 单人项目、刚接入 Cloudflare | 保持轻量。 | 控制台 + Wrangler + Git 记录变更。 |
| DNS、WAF、规则、Access 变多 | 纳入配置真源。 | 盘点现状，再按资源逐步接管。 |
| 多人维护生产域名 | 纳入配置真源。 | 入口和安全配置要有人复核。 |
| Worker 开发和部署 | 按需纳入。 | Wrangler 仍然是最直接的构建和部署入口。 |
| 已有大量控制台配置 | 导出后迁移。 | 导出现状，复核后再应用。 |
| 团队已有基础设施配置工具 | 可以沿用。 | 同一资源只保留一个配置真源。 |

## 工具用途

| 工具 | 用途 |
| --- | --- |
| 控制台 | 早期试错、低风险配置。 |
| Wrangler | Worker 本地开发、构建、部署、D1 迁移、日志。 |
| 基础设施配置工具 | DNS、规则、WAF、Access、账号和域名级资源。 |
| 生成/导入工具 | 把已有配置变成迁移草稿。 |
| 远程状态 | 团队共享配置状态。 |

## 优先纳入范围

| 顺序 | 资源 | 原因 |
| --- | --- | --- |
| 优先纳入 | DNS 记录、SSL/TLS、跳转 | 入口层改错影响最大。 |
| 优先纳入 | WAF、限流、Access | 安全策略需要复核和回滚。 |
| 再纳入 | 缓存规则、回源规则、改写规则 | 会影响缓存、回源、URL 和成本。 |
| 再纳入 | R2、KV、D1、Queues 等资源声明 | 资源归属要清楚；数据内容不进配置状态。 |
| 最后看 | Worker 发布流程 | 只有团队需要统一发布流程时再上。 |

## 迁移做法

盘点现状，生成草稿，人工复核，小步应用。被配置真源管理的资源，控制台只用于查看和排障。DNS、WAF、规则、Access 这类低频高风险配置优先纳入；Worker 代码、D1 迁移、临时排障和日志继续用 Wrangler / 控制台。
