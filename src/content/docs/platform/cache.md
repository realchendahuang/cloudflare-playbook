---
title: 缓存与 CDN
---

## 核查问题

| 问题 | 判断依据 |
| --- | --- |
| 资源是否公开静态文件？ | 图片、CSS、JS、字体、下载文件优先缓存。 |
| URL 变了是否代表内容变了？ | 带 hash 的文件可以长缓存；不带版本号用短缓存或重新验证。 |
| 响应里有没有登录态、用户资料或后台数据？ | 默认绕过共享缓存。 |
| HTML 和接口真的适合缓存吗？ | 要缓存就用缓存规则明确范围。 |
| 内容更新后的失效方式 | 优先改文件名，其次按 URL 精确清缓存。 |

## 推荐策略

| 资源 | 推荐策略 |
| --- | --- |
| 带版本号的静态文件。 | 长缓存。 |
| 图片、字体、公开下载文件。 | 边缘长缓存；大文件放 R2 自定义域名。 |
| HTML 页面。 | 短缓存、重新验证，或交给静态托管默认策略。 |
| 公开接口且低频变化。 | 用缓存规则限定路径和短缓存时间。 |
| 登录、后台、购物车、用户资料。 | 绕过共享缓存。 |

## 清缓存方式

优先用版本化文件名，其次按 URL 精确清理。全站清理只在误缓存用户数据、重大回滚或规则严重错误时用；它会让后续请求集中回源。

## 后面再看的缓存功能

| 能力 | 判断依据 |
| --- | --- |
| 分层缓存 | 有 VPS、对象存储或 R2 自定义域名源站时值得开启。 |
| R2 自定义域名 | 公开文件走 R2 + 缓存规则，比塞进站点包更合理。 |
| Cache Reserve | 单独付费；稳定大文件、源站出站带宽贵、提高缓存率能省钱时再算。 |
| Worker 自定义缓存 | 少数自定义响应才使用；静态资源优先使用标准缓存。 |

## 配置问题

| 常见做法 | 风险问题 |
| --- | --- |
| `Cache Everything` 覆盖整站。 | 可能把登录态、后台或预览内容放进共享缓存。 |
| 没有版本化 URL，只靠清缓存发版。 | 缓存状态更难预测。 |
| 把 CDN 当对象存储。 | 大文件、视频、下载归档看 R2 / Stream / Images。 |
| 让查询参数、登录状态和请求头随意影响缓存。 | 缓存率会被打碎，也更难排查。 |
| 先开 Cache Reserve 再算账。 | 它会产生存储和操作费用。 |
