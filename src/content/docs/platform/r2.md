最后核对日期：2026-06-18。

R2 是 Cloudflare 的 S3 兼容对象存储。它最适合保存文件、图片、用户上传、导出物、备份、日志归档和数据湖原始对象。R2 的核心吸引力是没有 egress bandwidth charge，但它仍然会按存储容量和 Class A / Class B 操作计费。

R2 的正确用法不是“把所有东西都扔进廉价存储桶”。对象本体放 R2，关系数据和可查询 metadata 放 D1，配置和低频缓存放 KV，异步处理放 Queues。这样既省钱，也不会把文件系统、数据库和任务队列搅在一起。

## 一句话判断

| 场景 | 是否优先用 R2 | 判断 |
| --- | --- | --- |
| 用户上传的图片、附件、导出文件 | 是 | 对象存储就是为文件本体准备的。 |
| 公开下载的大文件 | 是 | 无 egress bandwidth charge，但要估算 Class B 读取次数。 |
| 站点构建产物 | 通常否 | 文档站、官网、SPA 优先 Workers Static Assets 或 Pages。 |
| 图片原图 | 是 | 原图放 R2，需要动态裁剪和格式转换时再接 Images。 |
| 视频产品 | 不优先 | 视频存储、编码和播放优先看 Stream。 |
| 评论、订单、文章列表 | 否 | 关系查询、排序、筛选和事务放 D1。 |
| 配置、功能开关、缓存 | 否 | 读多写少的小值放 KV。 |
| 数据湖原始文件 | 可以 | 原始对象进 R2；查询分析再看 R2 Data Catalog、R2 SQL、Pipelines 或外部数仓。 |

## 二次精读结论

R2 的关键不是“免费文件硬盘”，而是和 Cloudflare CDN、Workers、Queues、D1、Images、Access 组合起来的对象层。

| 维度 | 结论 | 普通项目怎么落地 |
| --- | --- | --- |
| 成本 | R2 不收 egress bandwidth charge，但按 storage、Class A、Class B 和 IA retrieval 计费。 | 免费阶段先用 Standard；公开下载重点盯 `GetObject` / `HeadObject` 次数。 |
| 架构 | R2 存对象本体，不负责关系查询和业务索引。 | D1 存文件记录、权限、状态；R2 key 只做对象地址。 |
| 公开访问 | 生产公开资源走 custom domain；`r2.dev` 只适合测试。 | 要用 WAF、Access、Bot、缓存和 URL 分析，就必须用自定义域名。 |
| 一致性 | R2 本身强一致，Cloudflare Cache 会放宽一致性。 | 私有强一致读走 Workers API / S3 API；公开静态资源走版本化 URL + cache。 |
| 上传 | 全球用户上传到远区域 bucket 时，可以用 Local Uploads。 | Local Uploads 无额外费用，但 jurisdiction bucket 不支持。 |
| 归档 | IA 存储便宜，但读取贵，且最短计费 30 天。 | 备份、历史日志适合；热门下载和短期临时文件不要进 IA。 |
| 治理 | Bucket locks 能防覆盖和删除，优先于 lifecycle。 | 审计日志、备份、合规资料可以加锁；普通临时文件不要默认加锁。 |

## 上线前问题

| 问题 | 为什么要问 | 推荐答案 |
| --- | --- | --- |
| 对象是公开还是私有？ | 公共 bucket 和私有 Worker 鉴权的缓存、安全路径不同。 | 公开资源用 custom domain + cache；私有文件用 Worker / Access / presigned URL。 |
| 下载成本怎么算？ | R2 egress 免费，但 `GetObject` / `HeadObject` 是 Class B。 | 热门下载用 CDN 缓存、版本化 key 和合理 `Cache-Control`。 |
| 文件列表从哪里来？ | `ListObjects` 是 Class A，且分页和目录体验都不是数据库。 | D1 存 metadata；KV / 构建产物存公开 manifest。 |
| 上传方在哪里？ | 跨区域上传会影响延迟和可靠性。 | 多地区用户上传时评估 Local Uploads；有驻留要求时优先 jurisdiction。 |
| 是否需要浏览器直传？ | presigned URL 需要 CORS、过期时间和文件类型约束。 | 限定 `Content-Type`，返回 `expiresAt`，精确配置 CORS origin。 |
| 是否有合规保留要求？ | lifecycle 删除和 bucket lock 会互相影响。 | 需要防删防覆盖才用 Bucket locks；普通清理用 lifecycle。 |

## 运行模型

R2 可以通过 Worker binding、S3 兼容 API、公开桶自定义域名等方式访问。普通项目最常见的是 Worker 做入口控制，R2 存对象本体，D1 存 metadata。

```text
客户端
  │
  ├─ 请求上传授权 / 查询文件列表
  ▼
Worker
  ├─ env.BUCKET.get(key)
  ├─ env.BUCKET.put(key, stream)
  ├─ env.DB 查询文件 metadata
  └─ 生成 presigned URL 或流式返回对象
        │
        ▼
R2 Gateway
  ├─ Metadata Service：对象 metadata 和强一致状态
  ├─ Tiered Read Cache：读取加速
  └─ Distributed Storage：加密保存对象数据
```

官方架构里，R2 Gateway 运行在 Cloudflare 网络边缘，metadata 层负责对象状态和强一致性，数据存储层负责加密后的对象内容。写入完成并返回成功后，后续读取应该能在全球立刻看到最新对象状态。

## 免费与付费边界

R2 的免费额度只适用于 Standard storage，不适用于 Infrequent Access storage。免费额度按月计算。

| 能力 | Free | 付费价格 | 实践判断 |
| --- | --- | --- | --- |
| Standard storage | 10 GB-month/month。 | $0.015/GB-month。 | 小型图床、附件和导出文件早期很够用。 |
| Infrequent Access storage | 无免费额度。 | $0.01/GB-month。 | 冷数据更便宜，但读取会收费，且最短计费 30 天。 |
| Standard Class A | 1M requests/month。 | $4.50/million requests。 | 写入、列举、复制、multipart、生命周期转 IA 都算这里。 |
| Standard Class B | 10M requests/month。 | $0.36/million requests。 | `GetObject` / `HeadObject` 是下载热点的主要成本。 |
| Infrequent Access Class A | 无免费额度。 | $9.00/million requests。 | 冷数据写入和管理操作比 Standard 更贵。 |
| Infrequent Access Class B | 无免费额度。 | $0.90/million requests。 | 冷数据被频繁读取时会失去成本优势。 |
| Infrequent Access data retrieval | 无免费额度。 | $0.01/GB。 | IA 适合“很少读”的归档，不适合热门下载。 |
| Egress bandwidth | 免费。 | 免费。 | 这只是带宽不收费，不代表读取操作免费。 |
| DeleteObject / DeleteBucket / AbortMultipartUpload | 免费。 | 免费。 | 删除不收费，但生命周期、写入和读取仍要估算。 |
| 未授权请求 | 不收费。 | 不收费。 | 官方说明 HTTP 401 Unauthorized 不计 R2 操作费用。 |

Cloudflare 会把用量向上取整到下一个计费单位：例如 1,000,001 次操作按 2 million 计，1.1 GB-month 按 2 GB-month 计，IA 读取 1.1 GB 按 2 GB 计。估算成本时不要只看平均数，也要看是否刚好跨过一个计费单位。

## 计费证据怎么看

R2 预算判断要把“对象大小、请求次数、存储类别、缓存命中”拆开。不要只看带宽。

| 用量 | 计费口径 | 怎么看 |
| --- | --- | --- |
| Storage | 以 GB-month 计，按 billing period 内每日 peak storage 平均。 | Dashboard / GraphQL 的 `r2StorageAdaptiveGroups`，看 `payloadSize`、`metadataSize`、`objectCount`。 |
| Class A | 写入、列举、复制、multipart、bucket 配置和 lifecycle transition。 | `r2OperationsAdaptiveGroups` 按 `actionType` 看 `PutObject`、`ListObjects`、`UploadPart`。 |
| Class B | 读取对象和对象 metadata。 | 重点看 `GetObject`、`HeadObject`，公开下载热点会集中在这里。 |
| IA retrieval | 读取或复制 Infrequent Access 对象时按 GB 计。 | IA 只适合冷数据；一旦频繁读，storage 便宜会被 retrieval 吃掉。 |
| Egress | R2 直接 egress 到互联网免费。 | 这不代表 Images、Workers AI、Logpush、Stream 等其他产品不计费。 |
| 迁移工具 | Super Slurper / Sippy 本身免费，但写入 R2 会产生 Class A 和 storage。 | 大对象 multipart 会产生多次 Class A；源云厂商可能另收读取/出站费。 |

线上排查看三层：

1. 单条路径：一次上传是否触发 `PutObject`、multipart、D1 写入、Queues 消息和 Images 处理。
2. Bucket 整体：Dashboard Metrics 看 operations、storage、uploadCount 和 request distribution。
3. 历史趋势：GraphQL Analytics API 的 `r2OperationsAdaptiveGroups` / `r2StorageAdaptiveGroups` 看 31 天内变化。

## 关键限制

| 限制 | 当前值 | 实践判断 |
| --- | --- | --- |
| 单 bucket 存储容量 | Unlimited。 | 成本和对象组织比容量上限更重要。 |
| 单 bucket 对象数 | Unlimited。 | 列举要分页，目录结构要靠 key prefix 设计。 |
| 每个账户 bucket 数 | 1,000,000。 | 普通项目通常按环境或产品拆 bucket，不要按用户拆。 |
| 单 bucket 自定义域名 | 100。 | 公共资源可以用 `assets.example.com` 这类稳定域名。 |
| bucket 管理操作 | 50/second/bucket。 | 创建、删除、配置 bucket 是管理面操作，不是高吞吐对象接口。 |
| Cloudflare REST API | 1,200 requests/5 minutes/account。 | 高吞吐对象读写用 Workers API 或 S3-compatible API。 |
| object key | 1,024 bytes。 | key 既要可读，也要短；避免把完整用户输入直接当 key。 |
| object metadata | 8,192 bytes。 | metadata 只放对象描述，不放业务正文。 |
| object size | 5 TiB per object，实际脚注为 4.995 TiB。 | 超大文件用 multipart upload。 |
| single-part upload | 5 GiB，实际脚注为 4.995 GiB。 | 浏览器直传大文件优先 presigned URL + multipart。 |
| multipart upload | 4.995 TiB，最多 10,000 parts。 | parts 状态要保存在客户端、D1 或 Durable Objects。 |
| 同一个 key 并发写入 | 1 write/second。 | 高频覆盖同一路径会返回 429；用版本化 key 更稳。 |
| r2.dev 公开访问 | 非生产用途，可能限速和 429。 | 生产公开文件使用自定义域名。 |

这些限制里，普通项目最容易踩的是三个：把 `r2.dev` 当生产域名、热门文件下载只看 egress 不看 Class B、多个请求反复覆盖同一个 object key。

## Class A / Class B 怎么理解

R2 操作分两类。Class A 通常会改变状态或列举对象，Class B 通常读取已有状态。

| 类型 | 常见操作 | 成本直觉 |
| --- | --- | --- |
| Class A | `PutObject`、`CopyObject`、`ListObjects`、`CreateMultipartUpload`、`UploadPart`、`CompleteMultipartUpload`、`LifecycleStorageTierTransition` | 写入、列举、复制、multipart 和生命周期转储都要谨慎。 |
| Class B | `GetObject`、`HeadObject`、`HeadBucket`、`GetBucketCors`、`GetBucketLifecycleConfiguration` | 下载和探测对象是否存在会集中在这里。 |
| Free operations | `DeleteObject`、`DeleteBucket`、`AbortMultipartUpload` | 删除本身免费，但删除前后的读取、列举和写入仍可能计费。 |

公开下载的成本常常不是带宽，而是 `GetObject` 次数。官方 pricing 页给过一个典型例子：大量小文件每天被读取 10,000,000 次，一个月会产生大量 Class B 计费。普通项目做资源分发时，要用 Cloudflare Cache、合理的 Cache-Control 和版本化文件名减少回源读取。

## 一致性与缓存

R2 对对象操作提供强全局一致性：

| 操作 | 一致性 | 实践判断 |
| --- | --- | --- |
| 写入后读取 | Strongly consistent。 | 写入成功后，全球读取应该能看到最新对象。 |
| 更新 metadata | Strongly consistent。 | metadata 适合放对象级描述。 |
| 删除对象 | Strongly consistent。 | 删除后读取应立刻返回不存在。 |
| 列举对象 | Strongly consistent。 | `list()` 应反映该时间点的对象集合。 |
| IAM / API key 权限变化 | Eventually consistent，可能到一分钟。 | 刚改 token 权限时，测试要留传播时间。 |
| 同 key 并发 PUT / DELETE | last writer wins。 | 需要确定版本时使用不可变 key 或 D1 记录当前版本。 |

要特别区分 R2 本身的一致性和 Cloudflare Cache。通过自定义域名公开对象并启用缓存后，缓存层可能继续返回旧对象、旧 404 或已删除对象，直到 TTL 到期、缓存被驱逐或主动 purge。需要强一致读取的路径走 Worker binding 或 S3 API；公开静态资源则用版本化 URL。

## 数据位置和上传性能

R2 bucket 创建时可以选择 Automatic、Location Hint 或 Jurisdiction。这个决定影响对象存储位置、合规边界和上传路径。

| 选项 | 作用 | 注意 |
| --- | --- | --- |
| Automatic | 默认推荐，按创建请求位置选择接近的可用区域。 | 普通项目默认足够。 |
| Location Hint | 指定主要访问区域偏好：`wnam`、`enam`、`weur`、`eeur`、`apac`、`oc`。 | 只是 best effort；删除后用同名 bucket 重建会沿用原位置。 |
| Jurisdiction | 保证对象存储在指定 jurisdiction。 | 当前支持 `eu` 和 `fedramp`；创建后不能修改。 |
| Local Uploads | 上传先写到客户端附近位置，再异步复制到 bucket。 | 无额外费用，只按正常 Class A 上传计费；jurisdiction bucket 不支持。 |

如果用 jurisdiction bucket，Workers binding 和 S3 endpoint 都要带 jurisdiction。S3 endpoint 形如 `https://<ACCOUNT_ID>.<JURISDICTION>.r2.cloudflarestorage.com`。Logpush 直连 R2 和 Super Slurper 对 jurisdiction bucket 有额外限制，要按官方页面单独核对。

## Workers API 习惯

R2 绑定写在 `wrangler.jsonc` 里：

```jsonc
{
	"r2_buckets": [
		{
			// binding 是 Worker 代码里访问 R2 bucket 的变量名。
			"binding": "BUCKET",
			// bucket_name 是 Cloudflare 账户里的真实 R2 bucket 名称。
			"bucket_name": "uploads"
		}
	]
}
```

本地 `wrangler dev` 默认使用本地 R2 存储。确实需要连真实 bucket 调试时，可以在 binding 里加 `"remote": true`，但不要把生产 bucket 当成本地草稿箱。

### 流式读取和写入

Worker 适合做鉴权、路由、metadata 和流式转发，不适合把大文件完整读进内存。

Workers API 容易误解的点：

| 细节 | 判断 |
| --- | --- |
| `get()` 条件失败 | 可能返回没有 `body` 的 `R2Object`，不要只判断是否为 `null`。 |
| `put()` 条件失败 | 返回 `null`，对象不会被写入。 |
| `list()` | 最多 1,000 entries，但可能因为 metadata 和内存压力返回更少；必须看 `truncated`。 |
| `delete()` | 一次最多删除 1,000 个 key；删除强一致，但 Cloudflare Cache 可能仍有旧对象。 |
| `etag` | HTTP response header 应用 `httpEtag`，它已经带引号。 |
| checksum | 每次 PUT 只能指定一种 checksum algorithm。 |
| `resumeMultipartUpload()` | 不验证 uploadId 是否存在，后续操作要处理 upload 不存在的错误。 |
| 本地开发 | `wrangler dev` 默认本地 R2；需要真实 bucket 时才用 remote binding。 |

```ts
export default {
	// fetch 根据请求方法读取或写入 R2 对象。
	async fetch(request, env) {
		const url = new URL(request.url);
		const key = normalizeObjectKey(url.pathname);

		// key 校验失败时直接拒绝，避免空 key 和路径穿越。
		if (key === null) {
			return new Response('Invalid object key', { status: 400 });
		}

		// GET 走流式读取，避免把对象完整放进 Worker 内存。
		if (request.method === 'GET') {
			const object = await env.BUCKET.get(key);

			if (object === null) {
				return new Response('Not found', { status: 404 });
			}

			const headers = new Headers();
			object.writeHttpMetadata(headers);
			headers.set('etag', object.httpEtag);

			return new Response(object.body, { headers });
		}

		// PUT 直接把请求 body 写入 R2，Worker 只做入口控制。
		if (request.method === 'PUT') {
			await env.BUCKET.put(key, request.body, {
				httpMetadata: request.headers,
			});

			return new Response('Stored', { status: 201 });
		}

		return new Response('Method not allowed', {
			status: 405,
			headers: { allow: 'GET, PUT' },
		});
	},
};

// normalizeObjectKey 把 URL pathname 转成安全的 R2 object key。
function normalizeObjectKey(pathname) {
	const key = decodeURIComponent(pathname).replace(/^\/+/, '');

	// R2 key 不能是空字符串，也不应该接受路径穿越片段。
	if (key === '' || key.includes('..')) {
		return null;
	}

	return key;
}
```

读取时优先使用 `object.httpEtag` 写 HTTP header，因为它已经带引号，适合直接给客户端缓存校验。`object.writeHttpMetadata(headers)` 可以把内容类型、缓存控制等 metadata 转成 HTTP headers。

### 列表分页

`list()` 默认最多返回 1,000 个对象，并且为了降低内存压力，有时会返回少于 `limit` 的对象。不要用 `objects.length` 判断是否结束，要看 `truncated`。

```ts
export default {
	// fetch 分页列出指定 prefix 下的对象，适合后台或构建任务。
	async fetch(request, env) {
		const prefix = new URL(request.url).searchParams.get('prefix') ?? '';
		let cursor;
		const objects = [];

		do {
			// truncated 才是是否还有下一页的判断依据。
			const page = await env.BUCKET.list({ prefix, cursor });
			objects.push(...page.objects);
			cursor = page.truncated ? page.cursor : undefined;
		} while (cursor);

		return Response.json(objects);
	},
};
```

列表是 Class A 操作。公开页面不要每次请求都扫 bucket；把文件目录放到 D1、KV manifest 或构建产物里更稳。

### Presigned URL

Presigned URL 适合给浏览器或客户端临时授权一个对象的一次操作。官方支持 `GET`、`HEAD`、`PUT`、`DELETE`，有效期从 1 秒到 7 天。

| 方式 | 适合 | 注意 |
| --- | --- | --- |
| Worker 代理上传 | 小文件、需要统一鉴权和审计 | 注意 Worker 请求大小限制和内存，不要把大文件读成完整 buffer。 |
| Presigned URL | 直传 R2、临时下载、移动端上传 | URL 是 bearer token；限制 `Content-Type`，并配置 CORS。 |
| Temporary credentials | 多个 S3 操作、标准 S3 客户端 | 适合批量或客户端需要完整 S3 SDK 的场景。 |

Presigned URL 面向 S3 API 域名，不是自定义域名。需要在自定义域名上做鉴权时，可以考虑 Worker 鉴权、Cloudflare Access 或 WAF token authentication。

## S3 兼容边界

R2 的 S3-compatible API 很适合复用 SDK、CLI、PicGo、Obsidian 同步、数据迁移工具，但它不是完整复制 AWS S3 的所有功能。

| 细节 | 判断 |
| --- | --- |
| region | R2 region 使用 `auto`；不支持配置的工具可用空值或 `us-east-1` 作为 alias。 |
| endpoint | 标准 endpoint 是 `https://<ACCOUNT_ID>.r2.cloudflarestorage.com`；jurisdiction bucket 要加 jurisdiction。 |
| ACL / Object tagging / Website hosting | 许多 S3 功能未实现或部分实现，不要把 AWS S3 的 bucket policy、ACL、website hosting 心智直接照搬。 |
| ListObjects | 支持，但官方建议新应用使用 `ListObjectsV2`。 |
| Presigned URL | 支持 `GET`、`HEAD`、`PUT`、`DELETE`；不支持 HTML form `POST` multipart upload。 |
| Custom domain | Presigned URL 不能用于 custom domain。 |

## 公开访问

R2 bucket 默认不公开。公开访问有两种方式：

| 方式 | 用途 | 判断 |
| --- | --- | --- |
| 自定义域名 | 生产公开资源 | 可以使用缓存、WAF、访问控制、Bot 管理和 URL 级分析。 |
| `r2.dev` | 开发测试 | 官方明确不建议生产使用，可能限速和 429，不要 CNAME 到它。 |

公开 bucket 不会在根路径列出目录内容。你需要自己生成文件索引，或通过 D1 / KV / Worker API 控制列表。

如果通过自定义域名接入 Cloudflare Cache，默认只缓存部分文件类型。希望所有对象都缓存时，需要用 Cache Rules 或 Cache Everything 类规则，并配合版本化文件名和 purge 策略。

## CORS

浏览器跨域访问 R2 时需要配置 CORS，尤其是 presigned URL 直传。AllowedOrigins 要精确匹配 `scheme://host[:port]`，不要写路径。CORS 规则更新后可能需要最多 30 秒传播；如果对象已经被 Cloudflare Cache 缓存，旧响应不会立刻带上新 CORS headers，需要刷新或 purge 缓存。

```bash
pnpm wrangler r2 bucket cors set uploads --file cors.json
pnpm wrangler r2 bucket cors list uploads
```

## 生命周期和存储类别

R2 有 Standard 和 Infrequent Access 两种存储类别。

| 存储类别 | 适合 | 不适合 |
| --- | --- | --- |
| Standard | 热数据、公开下载、频繁读取对象 | 很少读取的大型归档。 |
| Infrequent Access | 冷数据、备份、长期留存文件 | 热门下载、频繁读取、短期临时对象。 |

生命周期规则可以把对象删除，或把 Standard 对象转到 Infrequent Access。转到 IA 会产生 Class A 操作；IA 最短计费 30 天；生命周期不能直接把 IA 转回 Standard，需要通过 copy 改变存储类别。

生命周期规则最多 1,000 条。生命周期删除一般会在 `x-amz-expiration` 后 24 小时内发生；如果删除和转 IA 在同一个 24 小时窗口冲突，删除优先。Bucket locks 会优先于 lifecycle，锁定期内对象不能被删除或覆盖。

默认情况下，未完成的 multipart uploads 会在 7 天后 abort。项目里如果允许大文件上传，要保存 upload id、part number、etag 等状态，避免客户端断线后找不回 multipart 状态。

## Bucket locks

Bucket locks 用来防止对象在指定时间内被删除或覆盖。它适合备份、审计日志、合规归档，不适合临时上传目录。

| 能力 | 判断 |
| --- | --- |
| 规则数量 | 每个 bucket 最多 1,000 条 lock rules。 |
| 范围 | 可以按 prefix 配置，也可以无 prefix 应用于全 bucket。 |
| 保留时间 | 支持固定时长、保留到指定日期或无限期。 |
| 冲突处理 | 多条规则命中同一对象时，最严格、最长的保留要求生效。 |
| 与 lifecycle | Bucket lock 优先于 lifecycle；锁定期内 lifecycle 不能删除对象。 |
| 删除 bucket | 还有 bucket lock rules 时不能 empty bucket，需要先移除规则。 |

## Event Notifications

R2 Event Notifications 可以把对象创建和删除事件发到 Queues。

```text
R2 object-create / object-delete
  │
  ▼
Queues
  │
  ▼
Consumer Worker
  ├─ 生成缩略图
  ├─ 抽取 PDF 文本
  ├─ 更新 D1 metadata
  └─ 发送通知
```

每个 bucket 最多 100 条 notification rules。事件类型主要是 `object-create` 和 `object-delete`。Queues 的吞吐边界要单独估算，官方文档给出的队列吞吐是 5,000 messages/second；更高吞吐可以把规则拆到多个 queue。

## 观测与排障

| 问题 | 看哪里 | 判断 |
| --- | --- | --- |
| 下载成本为什么上涨？ | `r2OperationsAdaptiveGroups` 的 `GetObject` / `HeadObject`。 | 公开热点没有命中 CDN cache，或客户端频繁探测对象。 |
| 写入和迁移成本为什么上涨？ | `PutObject`、`UploadPart`、`ListObjects`、`LifecycleStorageTierTransition`。 | multipart、大批量迁移、生命周期转 IA 都会推高 Class A。 |
| 存储为什么上涨？ | `r2StorageAdaptiveGroups` 的 `payloadSize`、`metadataSize`、`objectCount`、`uploadCount`。 | 可能是未完成 multipart、临时文件未过期、对象版本化 key 没清理。 |
| 公开对象删除后还在 | Cloudflare Cache / `cf-cache-status`。 | R2 已强一致删除，但自定义域名缓存还没过期或 purge。 |
| CORS 改了没生效 | 请求是否带 `Origin`、CORS rule、Cache。 | CLI / curl 不带 `Origin` 看不到 CORS header；缓存对象要 purge。 |
| `r2.dev` 429 | 公开开发 URL 限速。 | 生产必须换 custom domain。 |
| S3 SDK 报 region / signature | S3 endpoint、`region: "auto"`、jurisdiction endpoint。 | R2 的 S3 兼容不是 AWS 区域名。 |

## 适合的组合

| 目标 | 推荐组合 |
| --- | --- |
| 用户附件 | Worker + R2 + D1 metadata + presigned URL |
| 公开资源下载 | R2 + custom domain + Cloudflare Cache + 版本化 key |
| 图片服务 | R2 原图 + Images 变换 + Worker 鉴权 |
| 文档站附件 | Workers Static Assets 放站点，R2 放大文件和下载资源 |
| 后台导出 | Worker / Workflows + R2 + D1 记录导出状态 |
| 上传后处理 | R2 Event Notifications + Queues + Worker |
| 数据湖原始文件 | R2 + Pipelines + R2 Data Catalog / R2 SQL |
| 私有文件 | Worker 鉴权或 Access + R2，不开启 `r2.dev` |

## 不要这样用

| 误区 | 更好的做法 |
| --- | --- |
| 把 R2 当 SQL 数据库。 | metadata 和查询索引放 D1，对象本体放 R2。 |
| 把 `r2.dev` 当生产下载域名。 | 生产使用自定义域名，并配置缓存和访问控制。 |
| 只看到 egress 免费，就忽略 Class B。 | 公开下载要估算 `GetObject` 次数，并尽量用 CDN 缓存。 |
| 大文件上传先读成 `ArrayBuffer`。 | 用 stream、presigned URL 或 multipart upload。 |
| 用同一个 key 承接高频覆盖。 | 使用版本化 key，D1 记录当前版本。 |
| 每个页面请求都 `list()` bucket。 | 文件列表进 D1、KV manifest 或构建索引。 |
| 把热数据放 Infrequent Access。 | 热门对象使用 Standard，冷归档再用 IA。 |
| 修改 CORS 后期待缓存对象立刻变化。 | 等待传播并 purge 已缓存对象。 |
| 开了 custom domain 但忘记关 `r2.dev`。 | 需要 Access / WAF / token 控制时，禁用 public development URL。 |
| 以为 lifecycle 能覆盖 bucket lock。 | Bucket lock 优先，锁定期内 lifecycle 不能删除对象。 |

## 后续精读

R2 后续应该和这些产品一起读：

1. Queues：上传后处理、缩略图、PDF 解析、通知和重试。
2. Durable Objects：大文件 multipart 状态、实时上传房间和强一致协调。
3. Images：R2 原图和图片变换、缓存、成本边界。
4. Stream：视频业务不要用 R2 裸扛完整播放链路。
5. R2 Data Catalog / R2 SQL：对象存储进入数据分析场景。
6. Access / WAF：私有 bucket、自定义域名和下载鉴权。
