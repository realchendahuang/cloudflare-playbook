## 适用场景

| 场景 | 处理方式 |
| --- | --- |
| 用户上传图片、附件、导出文件 | 优先使用 R2，数据库只存对象标识和状态。 |
| 公开下载、软件包、PDF、备份 | 适合，但要看读取次数。 |
| 图床、小文件工具、导出物 | 适合需要控制服务器和对象存储成本的独立项目。 |
| 原图存储 + 缩略图处理 | 原图进 R2，转换看 Images 或 Worker。 |
| D1 里想存大段内容、图片、二进制 | D1 存索引，R2 存对象。 |
| 前端构建产物 | Static Assets / Pages。 |
| 视频产品 | 完整播放体验看 Stream。 |
| 高频小对象读写 | 需要先设计缓存和索引。 |

## 成本指标

| 成本信号 | 判断依据 |
| --- | --- |
| 存储容量 | 小项目文件、图片、备份可以起步；长期堆文件要做生命周期和清理。 |
| 写入类操作 | 上传、写入、列举、生命周期等偏修改操作会增长。 |
| 读取类操作 | 读取、下载、查看对象信息等偏读取操作会增长，公开下载热点尤其要看。 |
| 出站流量 | R2 的优势是出站带宽不收费，但读写操作和存储仍要算。 |
| 供应商迁移 | R2 兼容 S3 API；对象 key、元数据和访问层分开设计。 |
| 低频访问存储 | 冷数据再用，热文件优先 Standard。 |

## 设计原则

| 判断项 | 处理方式 |
| --- | --- |
| 数据库只存引用 | D1 存对象标识、归属、大小、状态和权限。 |
| 对象 key 要稳定 | 不把业务逻辑写死在随机路径里，迁移或换桶时影响更小。 |
| 上传权限 | 用户上传需要经过鉴权和签名。 |
| 下载区分公开和私有 | 公开文件可走自定义域名；私有文件用签名 URL 或 Worker 鉴权。 |
| 大文件处理 | Worker 做鉴权和路由，尽量流式处理。 |
| 热点读要加缓存 | 公开下载和图片访问要考虑 CDN 缓存。 |
| 冷数据再考虑 IA | 低频访问存储用于冷数据，热文件用 Standard。 |
