最后核对日期：2026-06-18。

R2 是 Cloudflare 的对象存储，适合放文件、图片、附件、备份、导入导出结果和数据湖对象。

先记住一句：**R2 适合存对象，不适合当数据库，也不适合塞进前端构建产物里。**

## 先判断

| 场景 | 建议 |
| --- | --- |
| 用户上传图片、附件、导出文件 | 优先用 R2，数据库只存 key 和状态。 |
| 公开下载、软件包、PDF、备份 | 适合，但要看读取次数。 |
| 原图存储 + 缩略图处理 | 原图进 R2，转换看 Images 或 Worker。 |
| D1 里想存大 JSON、图片、二进制 | 不适合，D1 存索引，R2 存对象。 |
| 前端构建产物 | 不优先，Static Assets / Pages 更合适。 |
| 视频产品 | 不优先，完整播放体验看 Stream。 |
| 高频小对象读写 | 谨慎，先设计缓存和索引。 |

## 成本先看什么

R2 成本看三件事：存了多少、写了多少、读了多少。它的 egress 免费很有价值，但不代表对象存储本身完全免费。

| 成本信号 | 判断 |
| --- | --- |
| Storage | 小项目文件、图片、备份可以起步；长期堆文件要做生命周期和清理。 |
| Class A operations | 上传、写入、列举、生命周期等偏修改操作会增长。 |
| Class B operations | 读取、下载、head 等偏读取操作会增长，公开下载热点尤其要看。 |
| Egress | R2 的优势是出站带宽不收费，但读写操作和存储仍要算。 |
| Infrequent Access | 冷数据再用，热文件不要一开始默认放 IA。 |

完整数字见 [免费额度大全](/platform/free-paid/)。

## 设计原则

| 原则 | 判断 |
| --- | --- |
| 数据库只存引用 | D1 存 object key、owner、size、状态和权限。 |
| 上传要有权限边界 | 用户上传不要直接裸写公共 bucket。 |
| 下载区分公开和私有 | 公开文件可走自定义域名；私有文件用签名 URL 或 Worker 鉴权。 |
| 大文件不要进 Worker 内存 | Worker 做鉴权和路由，尽量流式处理。 |
| 热点读要加缓存 | 公开下载和图片访问要考虑 CDN cache。 |
| 冷数据再考虑 IA | Infrequent Access 适合低频访问，不适合热文件。 |

## 常见误区

| 误区 | 更好的做法 |
| --- | --- |
| R2 免 egress，所以完全免费。 | 存储、Class A、Class B 仍然会计费。 |
| 把 R2 当数据库。 | R2 存对象，查询和事务交给 D1。 |
| 把用户文件放进 Static Assets。 | 用户文件进 R2，部署产物才进 Static Assets。 |
| 所有文件都公开。 | 私有文件走 Worker 鉴权或签名 URL。 |
| 一开始就用 Infrequent Access。 | 先确认访问频率，热文件用 Standard。 |
| 靠 list 对象做业务查询。 | 把索引写到 D1，不要扫 bucket。 |
