---
title: 边缘规则
description: Cloudflare 边缘规则的取舍、旧规则迁移和常见误区。
---

边缘规则解决“请求到 Cloudflare 之后怎么处理”：跳转、改路径、改请求头、改回源、改缓存或做少量边缘设置。应用后端看 Workers，规则配额按域名或账号计划判断。

## 先用规则的场景

| 需求 | 优先看 |
| --- | --- |
| 旧 URL、旧域名、`www` / 根域名统一跳转。 | 单条跳转 / 批量跳转。 |
| 改路径、查询参数、请求头或响应头。 | 改写规则。 |
| 按路径调整 Cloudflare 设置。 | 配置规则。 |
| 某些路径要走不同端口或后端。 | 回源规则。 |
| 明确控制哪些内容缓存。 | 缓存规则。 |

跳转、改写、缓存和回源优先用规则。需要数据库、D1 / KV / R2、鉴权业务、队列、复杂日志或完整工程流程时，再上 Worker。

## 边界

| 做法 | 问题 |
| --- | --- |
| 新项目继续写旧 Page Rules。 | 新实现优先现代规则；旧规则只作为迁移对象。 |
| 为每个小路径写一条规则。 | Free 规则数有限，先按业务入口合并。 |
| 全站跳转、改写、挑战验证、缓存。 | 容易误伤证书验证、后台、接口、登录和预览环境。 |
| 以为 Workers Paid 会提高规则配额。 | 规则跟域名或账号计划走，不跟 Workers Paid 走。 |
| 模拟通过就当线上没问题。 | 模拟只能排一部分问题；线上复盘还要看请求编号、真实日志和时间窗口。 |

## Page Rules 迁移

旧 Page Rules 按目的迁移：跳转看跳转规则，缓存看缓存规则，回源看回源规则，改路径和头看改写规则。每次只动少量关键路径，观察后再删旧规则。

## Snippets 和 Workers

Snippets 是轻量脚本，Workers 是完整开发者平台。

| 需求 | 判断 |
| --- | --- |
| 简单改请求头、轻量跳转、维护页。 | Pro+ 可以看 Snippets。 |
| 需要 D1、KV、R2、Durable Objects、Queues。 | 直接看 Workers。 |
| 需要发布流程、版本和复杂业务。 | 直接看 Workers。 |
| 还在 Free 域名计划。 | Snippets 不可用，Workers Free 可以跑小接口。 |

## 常见组合

| 场景 | 推荐组合 |
| --- | --- |
| 文档站域名规范化。 | 单条跳转做 `www` / 根域名统一。 |
| 旧博客迁移。 | 少量规律路径用单条跳转或改写规则；大量静态映射用批量跳转。 |
| 静态站缓存。 | 缓存规则控制 HTML 和带版本号的静态资源。 |
| 接口后端分流。 | 改写规则调路径，回源规则调后端；复杂鉴权交给 Worker。 |
