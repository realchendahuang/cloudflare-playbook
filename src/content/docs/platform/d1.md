最后核对日期：2026-06-18。

D1 是 Cloudflare 托管的 Serverless SQL 数据库，适合放普通项目里的关系型数据：评论、表单、用户资料、订单、文章元数据、小后台和小型 SaaS 业务表。

先记住一句：**D1 是关系数据层，不是文件库、日志库或大型分析库。**

## 先判断

| 场景 | 建议 |
| --- | --- |
| 评论、表单、文章元数据 | 优先用 D1。 |
| 小型 SaaS 业务表 | 可以用，适合用户、项目、订单、设置。 |
| 文件、图片、附件 | 不放 D1，文件进 R2，D1 只存 key 和状态。 |
| 登录会话 | 可以低频使用；强撤销和高频校验要谨慎。 |
| 高频计数器、库存、排行榜 | 先看 Durable Objects 或分片设计。 |
| 大型分析、事件日志 | 不放 D1，看 Analytics Engine、R2 或数据管道。 |
| 单库接近上限 | 不硬撑，按租户、项目或实体拆库。 |

## 成本先看什么

D1 主要看三件事：读了多少行、写了多少行、存了多少数据。它不按空闲数据库收费，但缺索引查询、全表扫描和大批量写入会很快吃掉免费层。

| 成本信号 | 判断 |
| --- | --- |
| Rows read 增长快 | 常见原因是没有索引、分页方式不对、后台筛选扫全表。 |
| Rows written 增长快 | 评论、表单、Webhook、导入、批量更新和删除都算写入。 |
| Storage 增长快 | 表、索引和历史数据都占空间；文件和大 JSON 应放 R2。 |
| 单库接近边界 | 按租户、项目或业务域拆库，不要把所有数据塞进一个库。 |
| 恢复窗口不够 | Time Travel 不是长期备份，重要数据要导出到 R2 或外部备份。 |

完整数字见 [免费额度大全](/platform/free-paid/)。

## 最重要的坑

| 坑 | 怎么避开 |
| --- | --- |
| 只看返回行数，不看扫描行数。 | 常用 `WHERE`、排序和关联字段要建索引。 |
| 把文件或大字段放进 D1。 | R2 放文件，D1 放元数据。 |
| 把 Time Travel 当备份。 | 重要数据定期 export 到 R2 或外部备份。 |
| 生产、预览、本地共用一个库。 | local、staging、production 使用不同数据库。 |
| 大更新一次跑完。 | 大迁移、大删除、大导入分批执行。 |
| 开读复制就不用管一致性。 | 写后读、认证、订单确认优先走 primary 或会话一致策略。 |

## 设计原则

| 原则 | 做法 |
| --- | --- |
| 先按页面建表 | 先列列表页、详情页、后台筛选和登录查询，再定索引。 |
| 写入可重试 | 评论、表单、Webhook、支付回调要有唯一键或去重字段。 |
| 小库优先 | 租户、项目、区域或实体边界清楚时可以拆库。 |
| 迁移进 Git | schema 变化用 migration 文件管理。 |
| 数据分层 | D1 存关系数据，R2 存文件，KV 存公开配置和缓存，Queues 处理慢任务。 |

## 常见误区

| 误区 | 更好的理解 |
| --- | --- |
| D1 是传统数据库的无脑替代品。 | D1 更适合 Cloudflare 上的小型、可拆分关系数据。 |
| 查询次数少就一定便宜。 | 成本看 rows read / rows written。 |
| 一个库能撑所有租户。 | Paid 单库也有上限，早期就要设计拆分边界。 |
| Time Travel 等于长期备份。 | 它只是短期恢复窗口。 |
| 用 D1 存日志最省事。 | 日志和分析会快速放大写入与存储。 |
