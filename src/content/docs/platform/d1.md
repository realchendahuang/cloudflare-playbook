## 先判断

| 场景 | 建议 |
| --- | --- |
| 评论、表单、文章元数据 | 优先用 D1。 |
| 小型 SaaS 业务表 | 可以用，适合用户、项目、订单、设置。 |
| 文件、图片、附件 | 不放 D1，文件进 R2，D1 只存对象标识和状态。 |
| 登录会话 | 可以低频使用；强撤销和高频校验要谨慎。 |
| 高频计数器、库存、排行榜 | 先看 Durable Objects 或分片设计。 |
| 大型分析、事件日志 | 不放 D1，看指标分析、R2 或数据管道。 |
| 单库接近上限 | 不硬撑，按租户、项目或实体拆库。 |

## 成本先看什么

| 成本信号 | 判断 |
| --- | --- |
| 读取行数增长快 | 常见原因是没有索引、分页方式不对、后台筛选扫全表。 |
| 写入行数增长快 | 评论、表单、第三方回调、导入、批量更新和删除都算写入。 |
| 数据量增长快 | 表、索引和历史数据都占空间；文件和大段内容应放 R2。 |
| 单库接近边界 | 按租户、项目或业务域拆库。 |
| 恢复窗口不够 | 时间恢复只覆盖有限窗口，重要数据要导出到 R2 或外部备份。 |

## 常见坑

| 坑 | 怎么避开 |
| --- | --- |
| 只看返回行数，不看扫描行数。 | 常用 `WHERE`、排序和关联字段要建索引。 |
| 把文件或大字段放进 D1。 | R2 放文件，D1 放元数据。 |
| 把时间恢复当备份。 | 重要数据定期导出到 R2 或外部备份。 |
| 生产、预览、本地共用一个库。 | 本地、预览、生产使用不同数据库。 |
| 大更新一次跑完。 | 大迁移、大删除、大导入分批执行。 |
| 开读复制就不用管一致性。 | 写后读、认证、订单确认优先走主库或会话一致策略。 |

## 怎么设计

| 判断 | 做法 |
| --- | --- |
| 先按页面建表 | 先列列表页、详情页、后台筛选和登录查询，再定索引。 |
| 写入可重试 | 评论、表单、第三方回调、支付回调要有唯一键或去重字段。 |
| 小库优先 | 租户、项目、区域或实体边界清楚时可以拆库。 |
| 迁移进 Git | 数据结构变化用迁移文件管理。 |
| 数据分层 | D1 存关系数据，R2 存文件，KV 存公开配置和缓存，Queues 处理慢任务。 |
