最后核对日期：2026-06-18。

D1 是 Cloudflare 托管的 Serverless SQL 数据库，使用 SQLite 语义，适合放普通项目里的关系型数据：评论、表单、用户资料、订单、文章元数据、小后台和小型 SaaS 的业务表。

先记住一句：**D1 不是“便宜版大数据库”，而是适合拆成多个小数据库的关系数据层。** 它很适合和 Workers、Static Assets、R2、KV、Queues 组合，但不要把文件、日志、计数器和大型分析都塞进去。

## 该不该用

| 场景 | 判断 |
| --- | --- |
| 评论、表单、文章元数据 | 优先用 D1，有结构、有筛选、有排序。 |
| 小型 SaaS 业务表 | 可以用，用户、项目、订单、订阅、设置都适合。 |
| 文件、图片、附件 | 不要放 D1，文件本体放 R2，D1 只存 key、状态和索引字段。 |
| 登录会话 | 低频、需要 SQL 查询时可以；强撤销和高频校验要谨慎。 |
| 高频计数器、库存、排行榜 | 不优先，先看 Durable Objects、Queues 或分片设计。 |
| 大型分析、事件日志、OLAP | 不优先，先看 Analytics Engine、R2、Pipelines 或外部数仓。 |
| 单库接近 10 GB | 不要硬撑，按租户、项目、区域或实体拆库。 |

## 免费与付费边界

D1 的费用主要来自三件事：读了多少行、写了多少行、存了多少数据。它不按空闲数据库或容量单位收费，D1 数据访问本身也不收 egress。

| 能力 | Workers Free | Workers Paid | 实践判断 |
| --- | --- | --- | --- |
| Rows read | 5 million/day | First 25 billion/month included，超出 $0.001/million rows | 缺索引会把免费读额度吃得很快。 |
| Rows written | 100,000/day | First 50 million/month included，超出 $1.00/million rows | 批量导入、更新、删除和索引维护都会算写入。 |
| Storage | 5 GB total | First 5 GB included，超出 $0.75/GB-month | 表和索引都算存储。 |
| Databases per account | 10 | 50,000，可申请提高 | Paid 适合按租户或项目拆库。 |
| 单个数据库大小 | 500 MB | 10 GB，不能再提高 | 单库上限是架构边界，不是最后才处理的运维问题。 |
| Account 总存储 | 5 GB | 1 TB，可申请提高 | 多库模型要看账户总量。 |
| Time Travel | 7 days | 30 days | 这是恢复窗口，不是长期备份。 |
| 每次 Worker invocation 的 D1 查询数 | 50 | 1,000 | 小心列表页、聚合接口和 N+1 查询。 |

其他固定限制也要记住：SQL 查询最长 30 秒，单行 / 字符串 / BLOB 最大 2 MB，SQL statement 最大 100 KB，单次查询最多 100 个 bound parameters，`d1 execute` 导入文件最大 5 GB。

## 最重要的坑

| 坑 | 为什么危险 | 怎么避开 |
| --- | --- | --- |
| 只看返回行数，不看扫描行数 | D1 读计费看 rows read，缺索引时返回一行也可能扫描整表。 | 先给常用 `WHERE`、排序和关联字段建索引，再看 `meta.rows_read`。 |
| 把文件或大字段放进 D1 | 单行最大 2 MB，数据库上限也很快被吃掉。 | R2 放文件，D1 放元数据。 |
| 把 Time Travel 当备份系统 | Free 只有 7 天，Paid 只有 30 天；恢复是原地覆盖。 | 重要数据定期 export 到 R2 或外部备份。 |
| 生产、预览、本地共用一个库 | 误跑迁移或测试写入会直接污染生产数据。 | local、staging、production 使用不同数据库。 |
| 大更新一次跑完 | 查询最长 30 秒，大批量更新容易失败或阻塞。 | 大迁移、大删除、大导入分批执行。 |
| 以为 Read Replication 会加速写入 | 读副本只服务读请求，写请求仍然到 primary。 | 先优化 SQL 和索引；读多且用户分布远时再开读复制。 |

## 设计原则

| 原则 | 做法 |
| --- | --- |
| 先按真实页面建表 | 先列出列表页、详情页、后台筛选、登录查询，再决定索引。 |
| 让写入可重试 | 评论、表单、Webhook 和支付回调要有唯一键或业务去重字段。 |
| 小库优先 | D1 官方推荐横向拆成多个小数据库，例如 per-user、per-tenant、per-entity。 |
| 迁移进 Git | schema 变化用 migration 文件管理，不要只在 Dashboard 手点。 |
| 恢复前先导出 | Time Travel restore 会覆盖原库，并取消进行中的查询。 |
| Read Replication 只服务读场景 | 必须配合 Sessions API；写后读、认证、订单确认优先走 primary 或 bookmark。 |

## 和其他产品怎么选

| 目标 | 更适合的产品 |
| --- | --- |
| 关系型数据、筛选、排序、事务感 | D1 |
| 读多写少的配置、缓存、公开索引 | KV |
| 文件、图片、附件、备份文件 | R2 |
| 单实体强一致、房间状态、连接状态 | Durable Objects |
| 后台异步处理、重试、削峰 | Queues |
| 大规模事件分析和指标 | Analytics Engine / R2 / Pipelines |

一个普通项目可以这样拆：Workers 处理请求，D1 存关系数据，R2 存文件，KV 存公开配置和缓存，Queues 处理慢任务，Turnstile / WAF 保护写入口。

## 常见误区

| 误区 | 更好的理解 |
| --- | --- |
| D1 是传统数据库的无脑替代品。 | D1 更适合 Cloudflare 上的小型、可拆分关系数据。 |
| 查询次数少就一定便宜。 | 成本看 rows read / rows written，缺索引的少量查询也可能很贵。 |
| 一个库能撑所有租户。 | Paid 单库也只有 10 GB，早期就要设计拆分边界。 |
| 开了读复制就不用管一致性。 | Read Replication 是异步复制，Sessions API 才能保证会话内顺序一致。 |
| Time Travel 等于长期备份。 | 它只是 7/30 天恢复窗口；长期备份要导出。 |
| 用 D1 存日志最省事。 | 日志和分析数据会快速放大 rows written 与 storage，应该走专门的数据产品。 |
