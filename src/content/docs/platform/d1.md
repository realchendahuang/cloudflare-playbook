最后核对日期：2026-06-17。

D1 是 Cloudflare 的 Serverless SQL 数据库，使用 SQLite 的 SQL 语义，可以从 Workers、Pages Functions 和 Cloudflare REST API 访问。它适合普通项目里的关系型数据：用户、评论、订单、文章元数据、配置后台、小型 SaaS 业务表。

D1 的正确用法不是“把一个传统大数据库搬到边缘”，而是把关系型数据拆成更小、更清楚、更靠近 Cloudflare 运行模型的数据库。官方文档也强调，D1 适合横向拆成多个较小数据库，比如 per-user、per-tenant、per-entity。

## 一句话判断

| 场景 | 是否优先用 D1 | 判断 |
| --- | --- | --- |
| 评论、表单、文章元数据 | 是 | 有结构、有查询、有排序，D1 比 KV 更合适。 |
| 小型 SaaS 业务表 | 是 | 用户、项目、订单、订阅、设置等关系型数据适合 D1。 |
| 登录会话 | 可以 | 需要 SQL 查询和过期时间时可用 D1；高频短 session 也可结合 KV。 |
| 文件、图片、附件 | 否 | 文件本体放 R2，D1 只存 metadata 和对象 key。 |
| 高吞吐计数器 | 谨慎 | 单库是顺序处理查询，强高频写入要考虑 Durable Objects、Queues 或分片。 |
| 大型分析 / OLAP | 不优先 | Analytics Engine、R2 + Pipelines、外部数仓更合适。 |
| 单库超过 10 GB | 不适合单库硬扛 | Paid 单库上限 10 GB，应该按租户、用户、区域或实体拆库。 |

## 二次精读结论

D1 的关键不是“SQLite 托管版”，而是 Cloudflare 开发者平台里的关系数据层。它和 Worker、Static Assets、R2、KV、Queues 的分工要提前想清楚。

| 维度 | 结论 | 普通项目怎么落地 |
| --- | --- | --- |
| 成本 | D1 按 rows read、rows written 和 storage 计费，不按空闲容量或数据库数量计费。 | 评论、表单、小后台可以免费起步；上线后用 `meta.rows_read` 和 `meta.rows_written` 看真实成本。 |
| 架构 | 单个 D1 database 是顺序处理查询的实例，吞吐取决于查询耗时。 | 先优化索引和查询，再考虑按 tenant、project、entity 拆库。 |
| 数据边界 | D1 放关系数据，不放文件本体、大日志和二进制内容。 | R2 存文件，D1 存 key、状态、权限、索引字段。 |
| 一致性 | 默认所有查询走 primary；Read Replication 需要 Sessions API 才能把读分到 replicas。 | 写后读、认证、订单确认用 `first-primary` 或 bookmark；公开列表可用 `first-unconstrained`。 |
| 恢复 | Time Travel 始终开启，但恢复是原地覆盖，会取消进行中的查询。 | 生产恢复前先导出，记录 previous bookmark，长期备份导出到 R2。 |
| 观测 | Dashboard、GraphQL、`meta` 和 `wrangler d1 insights` 分别看不同层次。 | 开发时看单条 query `meta`，线上看数据库 metrics 和慢查询。 |

## 上线前问题

| 问题 | 为什么要问 | 推荐答案 |
| --- | --- | --- |
| 最常用的查询是什么？ | D1 的成本和吞吐都受扫描行数影响。 | 先列出列表页、详情页、后台筛选、登录查询，再补索引。 |
| 写入有没有幂等键？ | 重试写操作只有幂等时才安全。 | 评论、表单、Webhook 写入要有唯一键、请求 id 或业务去重字段。 |
| 生产、预览、本地是否分库？ | 误把 migration 跑到生产是数据库事故。 | local / staging / production 分不同 D1 database，并在 Wrangler 环境里显式绑定。 |
| 是否需要读复制？ | Read Replication 解决远距离读延迟，不解决写入瓶颈。 | 内容列表、商品页、公开资料适合；写后读、认证、支付不要盲目读 replica。 |
| 是否需要数据驻留约束？ | jurisdiction 只能创建数据库时设置。 | 有 EU / FedRAMP 要求时创建前决定；location hint 只是提示，不是保证。 |
| 恢复和长期备份怎么做？ | Time Travel 不是长期归档。 | Time Travel 用于 7/30 天内回滚，长期备份用 export + R2 / 外部备份。 |

## 数据模型

D1 的心智模型可以这样看：

```text
Worker / Pages Function
  │
  ├─ env.DB.prepare(...).bind(...).all()
  │
  ├─ env.DB.batch([...])
  │
  └─ env.DB.withSession(...)
        │
        ▼
      D1 database
        ├─ SQLite SQL 语义
        ├─ rows read / rows written 计费
        ├─ Time Travel 备份
        └─ 可选 Read Replication
```

D1 每个数据库背后是顺序处理查询的实例。吞吐量主要取决于查询耗时：官方文档给的粗略例子是，如果平均查询 1 ms，大约可达 1,000 queries/s；如果平均查询 100 ms，则大约是 10 queries/s。也就是说，D1 的核心优化不是“多堆机器”，而是减少慢查询、减少全表扫描、减少一次请求里的无意义查询。

## 免费与付费边界

| 能力 | Workers Free | Workers Paid | 实践判断 |
| --- | --- | --- | --- |
| Rows read | 5 million/day。 | First 25 billion/month included，超出 $0.001/million rows。 | 索引直接影响 rows read，不能只看返回行数。 |
| Rows written | 100,000/day。 | First 50 million/month included，超出 $1.00/million rows。 | 高频写入、批量导入和索引写入都要估算。 |
| Storage | 5 GB total。 | First 5 GB included，超出 $0.75/GB-month。 | 表和索引都算存储。 |
| Databases per account | 10。 | 50,000，可申请提高。 | Paid 很适合按租户或项目拆库。 |
| 单个数据库大小 | 500 MB。 | 10 GB，不能再提高。 | 超过单库上限要拆库，不要等到最后迁移。 |
| Account 总存储 | 5 GB。 | 1 TB，可申请提高。 | 多库模型要监控账户总存储。 |
| Time Travel | 7 days。 | 30 days。 | 它是恢复窗口，不是长期归档。长期备份导出到 R2。 |
| 每次 Worker invocation 的 D1 查询数 | 50。 | 1,000。 | API 聚合查询要控制数量，避免 N+1。 |
| SQL 查询最长时间 | 30 seconds。 | 30 seconds。 | 大迁移、大更新要分批。 |
| 单行 / 字符串 / BLOB 上限 | 2,000,000 bytes。 | 2,000,000 bytes。 | 大字段和二进制放 R2。 |
| SQL statement 长度 | 100 KB。 | 100 KB。 | 大导入文件要拆成多条小语句。 |
| Bound parameters | 100/query。 | 100/query。 | 批量写入时用 batch 分组。 |
| Import file size | 5 GB。 | 5 GB。 | 更大的导入文件先切分。 |
| Egress | 不对 D1 数据访问收取 egress。 | 不对 D1 数据访问收取 egress。 | 成本主要来自 rows read、rows written、storage 和 Worker CPU。 |

## 计费证据怎么看

D1 的账单不是按“返回多少行”粗略估算，而是按实际扫描和写入的行数计量。每次查询返回对象里的 `meta` 就是最直接的证据。

| 字段 | 含义 | 怎么用 |
| --- | --- | --- |
| `meta.rows_read` | 本次查询扫描的行数。 | 判断索引是否有效；全表扫描会让免费读额度消耗很快。 |
| `meta.rows_written` | 本次查询写入的行数。 | `INSERT`、`UPDATE`、`DELETE` 和索引维护都会影响写入成本。 |
| `meta.duration` / `meta.timings.sql_duration_ms` | SQL 执行耗时。 | 评估单库吞吐，定位慢查询。 |
| `meta.size_after` | 查询后数据库大小。 | 观察接近 500 MB / 10 GB 单库上限的风险。 |
| `meta.served_by_region` | 执行查询的数据库区域。 | 验证 Read Replication 和数据位置效果；本地开发可能为空。 |
| `meta.served_by_primary` | 是否由 primary 执行。 | 判断读请求是否真的走到 replica，或是否仍在 primary。 |
| `meta.total_attempts` | 查询总尝试次数。 | 排查自动重试和瞬时错误。 |

线上排查时看三层：

1. 单条查询：看 Worker Binding API 返回的 `meta`。
2. 数据库整体：Dashboard 的 D1 Metrics 看 read queries、write queries、rows read、rows written、latency、storage。
3. 历史趋势：GraphQL Analytics API 或实验性的 `wrangler d1 insights` 看 31 天内的查询和慢 SQL。

## 查询习惯

D1 查询的第一原则是 prepared statements。用户输入不能拼进 SQL 字符串。

```ts
export default {
	async fetch(request, env) {
		const url = new URL(request.url);
		const id = url.searchParams.get('id');

		// 使用 bind 绑定用户输入，避免 SQL injection。
		const user = await env.DB.prepare('SELECT id, email FROM users WHERE id = ?')
			.bind(id)
			.first();

		return Response.json(user);
	},
};
```

常用方法：

| 方法 | 用途 | 判断 |
| --- | --- | --- |
| `prepare().bind()` | 绑定参数。 | 处理所有用户输入时都应该使用。 |
| `all()` | 返回多行和 `meta`。 | 列表查询、调试 rows read/write。 |
| `first()` | 返回第一行或单列。 | 详情页、登录查用户。 |
| `run()` | 执行 INSERT/UPDATE/DELETE。 | 写入不需要返回多行时使用。 |
| `raw()` | 返回数组形式。 | 大量只读数据、内部处理时更轻。 |
| `batch()` | 一次发送多条 prepared statements。 | 减少往返，避免 N+1，但仍要控制数量。 |
| `exec()` | 直接执行 SQL 字符串。 | 只用于维护、迁移、一次性任务，不用于用户输入。 |

每次查询返回的 `meta` 很重要，尤其是：

- `rows_read`：查询扫描了多少行，影响读计费。
- `rows_written`：写入了多少行，影响写计费。
- `duration`：SQL 执行耗时，影响单库吞吐。

API 容易误解的点：

| 细节 | 判断 |
| --- | --- |
| `run()` 和 `all()` | 官方文档说明 `run()` 功能上等价于 `all()`，写操作时 `results` 为空。 |
| `bind()` 参数 | 目前支持 ordered `?NNN` 和 anonymous `?`，命名参数还不是当前用法。 |
| `undefined` | 不支持，传入会触发 `D1_TYPE_ERROR`；缺省值要显式转成 `null`。 |
| Boolean | 写入会转成 INTEGER，读出是 `0` / `1`，不要假设读回布尔值。 |
| 大整数 | JavaScript number 有 52-bit 精度边界，超大 int64 读回可能不精确。 |
| `exec()` | 可执行一条或多条 SQL，但不做参数绑定，安全性和性能都不如 prepared statement。 |
| `batch()` | 多条 statement 顺序、非并发执行；如果某条失败，会回滚整个 batch。 |

## 索引策略

D1 的成本和性能都受索引影响。一个没有索引的 `SELECT * FROM users WHERE email = ?`，即使只返回一行，也可能扫描整张表。

推荐顺序：

1. 先根据页面和 API 写出真实查询。
2. 给常用 `WHERE`、排序、关联字段建索引。
3. 用 `EXPLAIN QUERY PLAN` 验证是否命中索引。
4. 用返回对象的 `meta.rows_read` 观察成本变化。
5. 写入频繁的字段不要盲目建太多索引，因为索引也会增加写入和存储成本。

```sql
CREATE TABLE IF NOT EXISTS users (
	id INTEGER PRIMARY KEY AUTOINCREMENT,
	email TEXT NOT NULL UNIQUE,
	name TEXT NOT NULL,
	created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_users_created_at ON users(created_at);

EXPLAIN QUERY PLAN SELECT * FROM users WHERE email = 'hello@example.com';
```

多列索引要注意左前缀规则。比如 `(customer_id, transaction_date)` 可以服务 `customer_id = ?`，也可以服务 `customer_id = ? AND transaction_date = ?`，但不能很好服务只有 `transaction_date = ?` 的查询。

## 迁移与环境

D1 migrations 是 SQL 文件版本管理。`wrangler d1 migrations create` 创建迁移文件，`wrangler d1 migrations apply` 应用迁移。官方文档建议迁移时可以用 database name，避免 binding name 改动后误操作。

```bash
wrangler d1 create app-db
wrangler d1 migrations create app-db create_users
wrangler d1 migrations apply app-db --local
wrangler d1 migrations apply app-db --remote
```

实践建议：

- `migrations/` 进入 Git，不要只在 Dashboard 手点 schema。
- local、staging、production 使用不同数据库。
- 生产迁移前先在本地和 staging 跑一遍。
- 大规模 `UPDATE` / `DELETE` 分批跑，避免 30 秒查询限制。
- 使用 Drizzle 等 ORM 时，可以用 `migrations_pattern` 匹配嵌套迁移文件。
- binding name 和 database name 都能用于 migration 命令；官方提醒 database name 更稳定，binding name 可能改。
- 外键相关迁移和导入可以用 `PRAGMA defer_foreign_keys = on` 暂时延迟校验，但事务结束前必须恢复一致。

Wrangler 环境要显式声明不同 D1 binding：

```jsonc
{
	"env": {
		"staging": {
			"d1_databases": [
				{
					"binding": "DB",
					"database_name": "app-staging",
					"database_id": "<STAGING_UUID>"
				}
			]
		},
		"production": {
			"d1_databases": [
				{
					"binding": "DB",
					"database_name": "app-production",
					"database_id": "<PRODUCTION_UUID>"
				}
			]
		}
	}
}
```

## 数据位置

D1 创建数据库时可以选择 automatic、jurisdiction 或 location hint。这个决定影响数据运行和持久化的位置，尤其是有合规要求时。

| 选项 | 作用 | 注意 |
| --- | --- | --- |
| Automatic | 默认方式，D1 会尽量把 primary 放在接近创建请求的位置。 | 普通项目默认足够。 |
| Jurisdiction | 限制数据库只在指定 jurisdiction 内运行和存储。 | 当前支持 `eu` 和 `fedramp`；只能创建时设置，不能后补或修改。 |
| Location hint | 给 primary 位置一个偏好，比如 `wnam`、`enam`、`weur`、`eeur`、`apac`、`oc`。 | 只是提示，不保证精确落在目标位置；jurisdiction 优先级更高。 |

普通项目的判断：多数写入来自哪里，primary 就尽量靠近哪里；读用户全球分布很广时，再用 Read Replication。需要合规驻留时，先选 jurisdiction，再考虑 Regional Services 等更完整的数据本地化能力。

## 备份、导入和恢复

D1 的 Time Travel 默认开启，不需要手动开关。Free 可恢复到 7 天内，Paid 可恢复到 30 天内。恢复是 destructive operation，会原地覆盖数据库，并取消进行中的查询。

```bash
wrangler d1 time-travel info app-db
wrangler d1 time-travel restore app-db --timestamp="2026-06-17T10:00:00+00:00"
```

Time Travel 不是长期备份。需要长期保留时，导出 SQL 到 R2 或仓库外的备份系统。

```bash
wrangler d1 export app-db --remote --output=./database.sql
wrangler d1 export app-db --remote --output=./schema.sql --no-data
wrangler d1 export app-db --remote --output=./data.sql --no-schema
```

导入时只能导入 SQL，不是直接上传 `.sqlite3` 文件。已有 SQLite 文件要先用 `sqlite3 db.sqlite .dump > db.sql` 转成 SQL，并按官方要求移除不兼容的事务语句。

恢复和导入的上线清单：

| 动作 | 风险 | 做法 |
| --- | --- | --- |
| Time Travel restore | 原地覆盖数据库，并取消 in-flight queries。 | 恢复前导出当前状态，记录 restore 返回的 previous bookmark。 |
| 长期备份 | Time Travel 只有 7/30 天窗口。 | 用 `wrangler d1 export`，或 Workflows 定期导出到 R2。 |
| 导入 `.sqlite3` | D1 不能直接导入 raw SQLite 文件。 | 先用 `sqlite3 file.sqlite .dump > file.sql` 转成 SQL。 |
| SQL dump 导入 | `BEGIN TRANSACTION` / `COMMIT` 可能导致错误。 | 按官方导入要求移除不兼容语句。 |
| 大 SQL 文件 | `d1 execute --file` 最大 5 GB，单条 SQL 最大 100 KB。 | 切分文件和大 INSERT。 |
| Virtual table / FTS5 | 当前 export 不支持 virtual tables。 | 需要先删除 virtual tables，导出后再重建。 |
| 大整数 | JavaScript 52-bit 精度会影响超大整数读写。 | ID 优先用字符串或安全整数范围内的数值。 |

## Read Replication 和 Sessions API

Read Replication 用于降低读延迟和提高读吞吐。它会创建 read-only replicas，写请求仍然转发到 primary database instance。

要点：

- 使用 read replication 必须使用 D1 Sessions API，否则查询仍只走 primary。
- session 可以用 `first-unconstrained`、`first-primary` 或上一次返回的 bookmark。
- bookmark 用于保证同一个逻辑会话里的 sequential consistency。
- 写后马上读、认证、金融交易这类一致性敏感场景，优先从 primary 或带 bookmark 的 session 读。

普通项目不需要一开始就上 Read Replication。更好的顺序是：先把 SQL、索引、分页、缓存做好；当用户分布很广、读请求很多、primary 距离导致明显延迟，再启用。

Sessions API 的三种入口：

| 入口 | 含义 | 适合场景 |
| --- | --- | --- |
| `withSession()` / `first-unconstrained` | 第一条查询可以走任意可用实例，优先低延迟。 | 公开列表、商品浏览、文档元数据等不要求最新数据的读。 |
| `first-primary` | 第一条查询走 primary，拿到最新数据库版本。 | 写后读、登录、权限、订单确认。 |
| bookmark | 新 session 至少读到上一个 session 已看到的数据库版本。 | 浏览器会话、分页、用户连续操作。 |

可以用 `meta.served_by_region` 和 `meta.served_by_primary` 验证查询由哪个区域和 primary / replica 执行。本地 `wrangler dev` 里这些字段可能为空。

## 观测与排障

| 问题 | 看哪里 | 判断 |
| --- | --- | --- |
| 免费读额度为什么很快没了？ | 单条 query `meta.rows_read`、Dashboard rows read。 | 多半是缺索引、全表扫描或后台列表没有分页。 |
| 写额度为什么涨得快？ | `meta.rows_written`、索引数量、写入路径。 | 写入 indexed column 会维护索引；批量导入和 DDL 也可能产生读写。 |
| 数据库变慢或 overloaded | D1 Metrics、`queryBatchTimeMs`、Debug D1 error。 | 优先优化慢 SQL、减少并发写、分批处理或拆库。 |
| 是否可以重试？ | Debug D1 error list。 | 读查询官方会自动重试最多两次；写查询只在幂等时做指数退避和 jitter。 |
| 线上报错怎么收集？ | `catch (e)` 记录 `e.message`。 | 不要把完整用户输入、PII、token 写日志。 |
| 哪些 SQL 最慢？ | GraphQL Analytics API、实验性 `wrangler d1 insights`。 | `wrangler d1 insights` 输出和参数可能变化，适合诊断不适合做稳定接口。 |

## 适合的组合

| 目标 | 组合 |
| --- | --- |
| 评论系统 | Worker + D1 + Turnstile + Rate Limiting |
| 文档元数据 | Workers Static Assets + D1 + Pagefind |
| 小型 SaaS | Worker + D1 + R2 + KV |
| 文件管理 | Worker + R2 存文件 + D1 存 metadata |
| 后台任务 | Worker + Queues + D1 |
| 多租户应用 | 每租户一个 D1 database，或按 tenant_id 分表后再演进拆库 |
| 全球只读查询 | D1 Read Replication + Sessions API |
| 长期备份 | D1 export + R2 + Workflows |

## 不要这样用

| 误区 | 更好的做法 |
| --- | --- |
| 把图片、附件、日志全文塞进 D1。 | 文件进 R2，D1 只存 key、状态、索引字段。 |
| 没有索引就上线列表查询。 | 所有常用 `WHERE` 和排序字段先用 `EXPLAIN QUERY PLAN` 看一遍。 |
| 用字符串拼 SQL。 | 所有外部输入都用 `prepare().bind()`。 |
| 在循环里对每条记录查一次数据库。 | 用 JOIN、IN 查询或 `batch()`，避免 N+1。 |
| 把 Time Travel 当长期备份。 | Time Travel 是恢复窗口，长期备份要导出到 R2。 |
| 生产和测试共用一个数据库。 | local、staging、production 分库。 |
| 单库快满了才想拆。 | 一开始就按 tenant、project、region 或 entity 设计拆分边界。 |
| 只看 query 数，不看 rows read。 | D1 读计费看扫描行数，缺索引的单条查询可能比很多小查询更贵。 |
| 以为 Read Replication 会让写入更快。 | 它主要降低读延迟和增加读吞吐；写请求仍到 primary。 |
