---
title: Queues
description: Cloudflare Queues 适合处理哪些后台任务，以及免费阶段怎么避免误用。
---

Queues 是 Cloudflare 的托管消息队列，适合把慢任务从用户请求里移出去：前台先返回，后台再处理邮件、通知、第三方回调、文件处理、外部服务调用、爬取和批量写入。它解决异步和削峰；长流程看 Workflows。

## 先判断

| 场景 | 建议 |
| --- | --- |
| 用户提交后发邮件、通知、第三方回调 | 适合。 |
| R2 上传后生成缩略图、解析文件 | 适合，消息里只放对象标识。 |
| 调用有频率限制的外部服务 | 适合，用并发限制保护下游。 |
| 爬虫、截图、Browser Run 任务 | 适合，慢任务放到后台处理。 |
| 批量写日志、事件、索引 | 可以，先入队再批量写。 |
| 用户必须立刻拿到结果 | 不适合，保留同步请求或做任务状态查询。 |
| 多步骤、长时间、需要状态机 | 看 Workflows。 |
| 强一致房间、计数器、锁 | 看 Durable Objects。 |

## 成本先看什么

Queues 的免费层按操作次数计算，不按“任务条数”计算。一次成功的小任务通常会经历写入、读取、删除；重试、失败队列、大消息和空轮询都会继续消耗操作。

| 边界 | 判断 |
| --- | --- |
| 标准操作次数 | 按消息生命周期估算，不按任务条数估算。 |
| 消息保留时间 | Free 阶段必须保证及时消费。 |
| 消息大小 | 大文件、正文和二进制内容放 R2，消息只放对象标识和必要元数据。 |
| 计费粒度 | 大消息会按分片计操作，小消息也要考虑写、读、删。 |
| 失败处理 | 重试和失败队列会让操作数量变大，生产任务必须可观测。 |

## 最重要的坑

| 坑 | 做法 |
| --- | --- |
| 把 10,000 次操作/天当成 10,000 条任务。 | 按消息数量和生命周期一起估算。 |
| Free backlog 超过 24 小时。 | 监控 backlog 和 oldest message。 |
| 以为消息只会执行一次。 | 每条任务带 `jobId`，写入时做幂等。 |
| 没有失败队列。 | 生产任务默认配失败队列和告警。 |
| 消息里塞大内容。 | 文件、正文、大对象放 R2 或 D1。 |
| 主动拉取时频繁空转。 | 普通 Worker 任务优先让队列主动推送。 |

## 设计原则

| 原则 | 判断 |
| --- | --- |
| 生产端要薄 | 校验请求、生成任务 ID、写入队列，然后返回。 |
| 消费端要幂等 | 重复执行是正常情况。 |
| 消息要小 | 队列传任务引用，不传文件和大正文。 |
| 下游要限速 | 外部服务、邮件、数据库写入都要按承载能力控制并发。 |
| 失败要能看见 | 重试次数、失败队列、告警和人工处理路径要提前定。 |
| 长流程 | 多步骤审批、等待、人机交互优先看 Workflows。 |
