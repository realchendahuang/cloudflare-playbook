## 先判断

| 场景 | 做法 | 原因 |
| --- | --- | --- |
| 开源文档站、官网、博客 | Pages。 | Git 集成和预览部署内置。 |
| 纯前端应用 | Pages。 | 静态请求免费且不限量。 |
| 少量表单、第三方回调、轻接口 | Pages Functions。 | 少量动态路径直接放在 Pages 项目里。 |
| 静态站 + 大量 Worker 接口 | Workers Static Assets。 | 动态能力多时和 Worker 绑定放在同一项目。 |
| 用户上传、附件、下载包 | R2。 | Pages 只放构建产物。 |

## 免费阶段怎么用

| 项目 | 判断 |
| --- | --- |
| 静态资产 | 文档、官网、博客和前端 bundle 应尽量停在静态层。 |
| 动态函数 | 少量表单、第三方回调和接口可以起步，多绑定场景优先迁到 Workers Static Assets。 |
| 构建和预览 | 高频提交、大型构建、多分支预览会先撞到构建侧限制。 |
| 文件和附件 | Pages 只放构建产物，大文件、图片原图、上传和下载包放 R2。 |
| 预览安全 | 预览地址需要按敏感程度加 Access。 |

## 和 Workers Static Assets 怎么选

| 问题 | Pages | Workers Static Assets |
| --- | --- | --- |
| 部署来源 | Git / Pages 项目设置。 | Worker 项目配置。 |
| PR 预览 | 内置。 | 需要自己组织。 |
| 静态请求成本 | 免费且不限量。 | 免费且不限量。 |
| 动态能力 | Pages 动态函数，适合少量接口。 | Worker 原生入口，适合多绑定组合。 |
| 更适合 | 内容协作、官网、文档。 | 静态站 + 接口 / 数据 / AI。 |

## 动态函数

| 判断 | 做法 |
| --- | --- |
| 只有少量接口 | Pages 动态函数。 |
| 静态请求不该触发动态函数 | 控制动态路径，避免把整站阅读流量打进动态函数。 |
| 鉴权或后台路径 | 请求耗尽时倾向 fail closed。 |
| 已经需要多路由和多个绑定 | 迁到 Workers Static Assets。 |
