---
title: 平台化与多租户
description: 什么时候才需要客户域名、客户代码和平台化产品。
---

## 先判断

| 你要做什么 | 先用什么 | 边界 |
| --- | --- | --- |
| 文档站、官网、后台 | Workers / Static Assets / D1 / R2 / Access。 | 平台化产品放到客户域名或客户代码需求出现后。 |
| 多租户业务数据 | 租户模型、资源隔离、权限和审计。 | 客户变多时，先处理数据隔离和权限。 |
| 客户绑定自己的域名 | Cloudflare for SaaS / Custom Hostnames。 | 需要自动验证、证书和清理流程。 |
| 客户长期部署自己的代码 | Workers for Platforms。 | 需要隔离、日志、limits 和出站控制。 |
| 需要短期执行不可信代码 | Dynamic Workers 或 Sandbox 相关能力。 | 不可信代码要和主 Worker 隔离。 |

## 先问清楚

客户是否真的需要自己的域名，客户是否要上传或运行自己的代码，你是否已经能管理验证、证书、下线、租户限额、滥用处理和计费边界。

## 推荐路线

| 阶段 | 做法 |
| --- | --- |
| 早期 SaaS | 先用普通 Workers、D1、R2、KV 和 Access，把租户模型做清楚。 |
| 客户域名成为主要卖点 | 做 Custom Hostname onboarding、验证、证书状态和清理后台。 |
| 客户代码成为产品功能 | 再评估 Workers for Platforms，先设计隔离、限额、日志和出站控制。 |
| 不可信代码执行成为主要流程 | 把稳定标识、最小权限、预算和审计作为第一版需求。 |
